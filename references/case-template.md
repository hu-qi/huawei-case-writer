## Strong conventions extracted from the official template

- The article is instructional and task-oriented.
- The flow should be visible and easy to execute even if screenshots are unavailable.
- Menu clicks should be described with hierarchical paths.
- Core code, config, commands, or prompts should be explained, not dumped without context.
- Paid resources should be summarized and later released.
- Preparatory steps should gather coupons, tokens, tools, SDKs, access keys, or service activation up front.
- The final article should include cleanup steps and extension reading.

## Additional conventions from the pasted reference cases

- `案例流程` often contains a short explanatory paragraph before the numbered `说明` list.
- Skill-based cases usually include a section that explains the skill directory and the role of each file.
- AI dialogue cases usually include the exact prompt text in a code block.
- After a major action, the article usually states what the system automatically creates, loads, or displays.
- Case subsections are compact and aligned to one operation each.

## Formatting rules

- Use markdown headings with numbered hierarchy.
- Use code fences for shell, yaml, json, python, text prompts, and code snippets.
- Use tables for resources and parameter explanations.
- Use inline code for paths, filenames, parameters, class names, methods, and URLs when used as code artifacts.
- Use short paragraphs and avoid overly academic exposition.
